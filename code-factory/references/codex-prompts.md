# Codex Prompt 模板

本文件包含各阶段调用 Codex 时使用的 prompt 模板。
Claude 在调用 codex exec 时应参考这些模板，根据实际情况替换占位符。

---

## 1. 设计评审 Prompt

### 单维度评审（通用）

```
你是一名资深软件架构师，正在评审一份技术设计方案。

以下是设计方案全文：
---
{01-design.md 的内容，通过 cat 内联}
---

请从以下角度进行评审：
- 架构合理性与可维护性
- 实现方案的正确性和完整性
- 潜在的风险和遗漏

输出格式（严格遵守），写入 {session}/02-review/r1.md：

第一行：DECISION: APPROVE 或 DECISION: REQUEST_CHANGES
第二行：PRIORITY_ISSUES: <问题总数>
第三行：---（分隔线）

然后按以下结构输出：

## P0 问题（必须改）
每条不超过50字，直接说问题是什么。

## P1 问题（建议改）
每条不超过50字。

## 详细分析
对每个 P0/P1 问题展开说明：问题原因、影响范围、改进方案。

## 对待评审问题的回应
逐条回应设计文档末尾列出的待评审问题。
```

### 架构维度评审

```
你是一名资深软件架构师，专注于系统架构评审。

以下是设计方案：
---
{01-design.md 内容}
---

从【架构合理性与可维护性】角度评审，重点关注：
- 模块划分是否清晰，职责是否单一
- 依赖关系是否合理，是否存在循环依赖
- 扩展性设计是否充分
- 是否符合项目现有架构风格

输出格式同上，写入 {session}/02-review/r1-architecture.md
```

### 安全维度评审

```
你是一名安全工程师，专注于代码安全评审。

以下是设计方案：
---
{01-design.md 内容}
---

从【安全性与边界情况】角度评审，重点关注：
- 输入验证和数据清洗
- 权限控制和认证授权
- 敏感数据处理
- 常见漏洞风险（注入、XSS、CSRF等）
- 异常和错误处理

输出格式同上，写入 {session}/02-review/r1-security.md
```

### 性能维度评审

```
你是一名性能工程师，专注于系统性能评审。

以下是设计方案：
---
{01-design.md 内容}
---

从【性能与可扩展性】角度评审，重点关注：
- 时间复杂度和空间复杂度
- 数据库查询效率
- 缓存策略
- 并发处理
- 瓶颈和热点

输出格式同上，写入 {session}/02-review/r1-performance.md
```

## 2. 讨论修订 Prompt

### Codex 重新评审（增量）

```
你是上一轮的评审者。以下是修订后的设计方案：
---
{01-design-v{N}.md 内容}
---

设计者对以下问题进行了修改：
{Claude 列出的修改点和修改理由}

请只评审上述修改点，不重复已通过的部分。
如果修改满意，可以 APPROVE；如果仍有问题，继续 REQUEST_CHANGES。

输出格式同前，写入 {session}/02-review/r{N}.md
```

## 3. 任务拆分 Prompt

```
你是一名技术项目经理，擅长将设计方案拆分为可执行的开发任务。

读取以下文件：
---
{01-design-final.md 内容}
---

评审共识：
---
{02-review/consensus.md 内容}
---

将实现拆分为独立、可并行的开发任务。要求：
1. 每个任务职责单一，可独立实现和测试
2. 明确标注任务间的依赖关系
3. 标注复杂度和涉及的文件
4. 划分执行波次（同一波次内的任务可并行）

输出格式（严格遵守），写入 {session}/03-plan.md：

## TASKS
- TASK:{kebab-case-id} DEPENDS:{none|逗号分隔的task-id} COMPLEXITY:{low|medium|high} FILES:{逗号分隔的文件路径}
  {一句话描述该任务}

## WAVES
- WAVE-1: {逗号分隔的task-id列表}
- WAVE-2: {逗号分隔的task-id列表}

## ACCEPTANCE
- [ ] {验收标准}
```

## 4. 代码实现 Prompt

```
你是一名高级软件工程师。

任务：{Task 描述}
涉及文件：{file-paths}

设计要求：
---
{从 design 中提取的与该 Task 相关的部分}
---

请实现所有代码变更，并编写对应的测试用例。
要求：
- 代码风格与项目现有代码保持一致
- 注释语言与现有代码库保持一致
- 如果涉及新文件，放在合理的目录结构下

完成后写摘要到 {session}/04-impl/task-{id}-summary.md

摘要格式：
第一行：STATUS: DONE 或 STATUS: PARTIAL
第二行：FILES_CHANGED: {逗号分隔的文件路径}
第三行：---
然后简述：做了什么、修改了哪些逻辑、遇到了什么问题（如有）
```

## 5. 代码 Review Prompt

### 全量 Review

```
你是一名代码评审专家。

实施计划：
---
{03-plan.md 内容}
---

请审查以下文件的代码变更：
{FILES_CHANGED 列表}

检查维度：
- 功能正确性：是否完整实现了设计要求
- 边界情况：空值处理、异常处理、并发安全
- 代码质量：命名规范、结构清晰、无重复代码
- 测试覆盖：核心路径是否有测试覆盖

输出格式，写入 {session}/05-review/codex-review.md：

第一行：VERDICT: PASS 或 VERDICT: ISSUES_FOUND
第二行：CRITICAL_ISSUES: <数字>
第三行：---

## 关键问题
{列出需要修复的问题，每条包含：文件、行号、问题描述、修复建议}

## 改进建议
{非阻断性的改进建议}

## 总体评价
{一段话总结代码质量}
```

### 分模块 Review（大需求）

```
你是一名代码评审专家，负责审查 {模块名} 模块。

该模块的设计要求：
---
{从 plan 中提取的该模块 Task 描述}
---

请审查以下文件：
{该模块的 FILES_CHANGED}

检查维度和输出格式同上，写入 {session}/05-review/codex-review-{module}.md
```

## 6. 修复 Prompt

```
你是修复工程师。以下是代码评审中发现的关键问题：

{从 codex-review.md 中提取的关键问题列表}

请逐一修复这些问题。修复时：
- 保持原有设计意图不变
- 只修改必要的部分，不做额外重构
- 如果修复某个问题会引入新的影响，请在摘要中说明

完成后写摘要到 {session}/04-impl/fix-r{N}-summary.md

摘要格式：
STATUS: DONE 或 PARTIAL
FILES_CHANGED: {文件路径}
---
{每个修复的简述}
```
